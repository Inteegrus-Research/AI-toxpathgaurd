# ToxPathGuard — Multi-Pathway Molecular Safety Audit

> A fully offline, single-machine molecular pre-screening prototype that evaluates a compound across three independent biological stress pathways and explains its reasoning.

**Built for the Codecure AI Hackathon at IIT (BHU).**

---

## Overview

ToxPathGuard takes a SMILES string as input and runs it through three separate XGBoost classifiers, each trained on a different Tox21 stress-response assay. The system produces a per-pathway risk probability, a final verdict, SHAP-based feature explanations, structural toxicophore alerts, and an applicability-domain warning — all displayed in a clean Streamlit dashboard that runs entirely offline.

This is a decision-support prototype for early-stage compound safety review. It is not a clinical diagnostic tool, it does not replace wet-lab assays, and its predictions should not be used for regulatory or medical decisions.

---

## Problem Statement

Early-stage drug discovery generates thousands of candidate compounds that need to be triaged for obvious safety liabilities before expensive synthesis or testing begins. Computational toxicity prediction can catch clear red flags early — but most simple classifiers behave like black boxes: they return a single score with no explanation, no uncertainty estimate, and no indication of which biological mechanism is being triggered.

The result is that a researcher gets a number but not an answer. ToxPathGuard is built around the idea that a more useful tool should tell you *which stress pathway* is flagged, *why* the model thinks so, *which chemical features* are driving the score, and *how much to trust the result* given whether the molecule looks like anything in the training data.

---

## What the System Does

Given any valid SMILES string, ToxPathGuard:

1. Validates and parses the molecule using RDKit.
2. Generates a 1030-dimensional feature vector (Morgan fingerprint + 6 physicochemical descriptors).
3. Runs the molecule through three independent XGBoost binary classifiers — one per biological stress pathway.
4. Applies per-assay safety thresholds calibrated to minimise false negatives.
5. Screens the molecule against a panel of 11 structural toxicophore SMARTS patterns.
6. Checks whether the molecule falls within the chemical space of the training data using Tanimoto similarity.
7. Runs a rule-based coordinator that combines all signals into a final verdict (Cleared / Caution / High Risk) with a confidence level and a human-readable reason summary.
8. Computes SHAP values for each model to explain which features drove the prediction.
9. Returns everything in a structured output that the Streamlit UI renders as a clean, scrollable audit report.
10. Lets you download the full audit as a JSON report.

---

## Key Features

**Three independent biological stress checkpoints.** Rather than a single toxicity score, ToxPathGuard screens each molecule against the DNA/genotoxic stress pathway (SR-p53), the oxidative stress pathway (SR-ARE), and the cellular heat-shock stress pathway (SR-HSE) separately. This lets the system flag *which* mechanism is most concerning, not just *whether* a molecule is risky.

**Structural toxicophore detection.** An RDKit SMARTS-based alert panel screens for known high-risk chemical patterns — aromatic amines, nitro groups, quinones, PAH scaffolds, Michael acceptors, epoxides, N-nitroso groups, and others. The system classifies the relationship between alert matches and model scores (Confirmed Toxicophore, Novel Risk Motif, Alerted but Model-Low) rather than just listing alerts.

**Applicability-domain checking.** The system computes Tanimoto similarity between the query molecule and a stratified sample of training fingerprints. If a molecule is structurally far from anything seen during training, the output explicitly flags this as a low-confidence extrapolation.

**SHAP-based explanations.** Every prediction comes with exact SHAP values computed via `shap.TreeExplainer`. The output separates fingerprint contributions (which structural patterns drove the score) from physicochemical descriptor contributions (LogP, TPSA, molecular weight, etc.), making the explanation accessible to both ML practitioners and chemists.

**Safety-oriented threshold policy.** The default operating mode uses sensitivity-optimised thresholds rather than the F1-maximising thresholds from training, because a pre-screening tool should miss fewer toxic compounds even at the cost of more false alarms. Both threshold sets are visible in the output.

**Rule-based coordinator.** A deterministic coordinator reads all agent outputs, alert results, and domain status to produce a final verdict, a confidence level (High/Medium/Low), and a short natural-language reasoning summary. Every sentence in that summary traces back to a specific decision rule — nothing is generated by a language model.

**Fully offline.** No API calls, no cloud services, no internet connection required after setup. All models, data, and logic live in a single local project folder.

## Tech Stack

The system is built entirely in Python. The molecular parsing, feature extraction, 2D rendering, and structural alert matching all use RDKit. The three classifiers are XGBoost models trained with `scale_pos_weight` to handle class imbalance. SHAP values are computed with the `shap` library's `TreeExplainer`, which is exact (not sampled) for tree-based models. The Streamlit app is the only user-facing component.

| Component | Library |
|---|---|
| Molecule parsing & features | RDKit |
| Classification | XGBoost |
| Explainability | SHAP |
| UI | Streamlit |
| Data handling | pandas, numpy |
| Structural alerts | RDKit SMARTS |
| Domain check | NumPy (Tanimoto) |
| Report export | Python json |

## System Architecture

The design follows a strict layered pipeline. The UI calls exactly one function from `src/predict.py`, which orchestrates everything else. No logic is duplicated in the app layer.

```
SMILES input
    │
    ▼
src/features.py       ← RDKit parsing + 1030-dim feature vector
    │
    ├──▶ src/alerts.py     ← SMARTS structural alert screen
    ├──▶ src/domain.py     ← Tanimoto applicability-domain check
    │
    ├──▶ agent_p53.pkl     ← XGBoost: SR-p53 (DNA/genotoxic stress)
    ├──▶ agent_are.pkl     ← XGBoost: SR-ARE (oxidative stress)
    ├──▶ agent_hse.pkl     ← XGBoost: SR-HSE (cellular/heat-shock stress)
    │
    ├──▶ src/explain.py    ← SHAP TreeExplainer per agent
    │
    ▼
src/reasoner.py       ← deterministic coordinator: verdict + confidence + summary
    │
    ▼
app/streamlit_app.py  ← display only; no prediction logic
    │
    ▼
src/report.py         ← JSON export
```

## Installation

Clone the repository and create a virtual environment. Python 3.10 or 3.11 is recommended.

```bash
git clone https://github.com/inteegrus-research/ai-toxpathgaurd.git
cd ai-toxpathguard
python -m venv venv
source venv/bin/activate          # on Windows: venv\Scripts\activate
pip install -r requirements.txt
```

The `requirements.txt` includes: `rdkit`, `xgboost`, `shap`, `streamlit`, `pandas`, `numpy`, `scikit-learn`, `matplotlib`, `Pillow`.

If you are on Arch Linux and using a system Python, add `--break-system-packages` to the pip command or use the virtual environment as shown above.

---

## How to Run

**Step 1 — Dataset audit** (verify the Tox21 data before anything else):
```bash
python src/audit.py --data data/tox21.csv
```

**Step 2 — Feature engineering** (generates the .npz files in `data/processed/`):
```bash
python src/features.py --data data/tox21.csv --out data/processed
```

**Step 3 — Model training** (trains three XGBoost models, writes to `models/`):
```bash
python src/train.py --processed data/processed --out models
```

**Step 4 — Verify the prediction layer works**:
```bash
python src/predict.py --smiles "CCO"
python src/predict.py --smiles "c1ccc2c(c1)cc1ccc3cccc4ccc2c1c34"
```

**Step 5 — Run the demo test suite** (pre-demo stability check):
```bash
python tests/run_demo_tests.py --verbose
```

**Step 6 — Launch the app**:
```bash
streamlit run app/streamlit_app.py
```

The app opens at `http://localhost:8501` and works without any internet connection.

## The Three Biological Stress Pathways

**SR-p53 — DNA / Genotoxic Stress.** The p53 tumour suppressor protein is activated in response to DNA damage. A compound that activates the SR-p53 reporter in Tox21 is flagged as having potential genotoxic activity — meaning it may directly or indirectly damage DNA or interfere with DNA repair.

**SR-ARE — Oxidative Stress (Nrf2/ARE pathway).** The antioxidant response element (ARE) is activated when a cell detects elevated reactive oxygen species or electrophilic stress. Compounds that activate SR-ARE may cause oxidative damage, disrupt redox homeostasis, or react covalently with cellular proteins and DNA.

**SR-HSE — Cellular / Heat-Shock Stress.** The heat-shock element (HSE) reporter is activated by the unfolded protein response and general cellular stress. Activation suggests the compound is causing broad protein damage, membrane disruption, or cellular stress beyond what is captured by the other two pathways.

These three pathways are mechanistically distinct. A compound can fail one, two, or all three checkpoints independently.

## Threshold and Explainability Notes

The system uses two threshold sets. The **training threshold** for each assay is the F1-maximising operating point identified during training (0.76 for SR-p53, 0.48 for SR-ARE, 0.80 for SR-HSE). The **safety threshold** is a lower, sensitivity-oriented cutoff (0.35 / 0.20 / 0.35) that reduces false negatives at the cost of more false positives — the appropriate trade-off for a pre-screening tool. The default mode in the UI is safety thresholds; the sidebar toggle lets you compare both.

SHAP values are computed using `shap.TreeExplainer`, which is exact for XGBoost and runs in a few milliseconds per sample. A positive SHAP value means that feature pushed the model's score toward "toxic"; a negative value means it pushed toward "non-toxic". The descriptor contributions (LogP, TPSA, MolWt, HBD, HBA, RingCount) are always separately surfaced because they are chemically interpretable, while the Morgan fingerprint bit contributions indicate which structural substructures were most influential.

## Limitations

The models are trained on Tox21, which is a well-curated public dataset but is modest in size (roughly 5,800–6,800 usable samples per assay after SMILES cleaning and missingness filtering). The positive class is sparse — 5.8% to 16.2% across the three assays — which limits the classifiers' precision regardless of how well imbalance is handled. The PR-AUC values on the held-out test sets are 0.387 (SR-p53), 0.520 (SR-ARE), and 0.383 (SR-HSE), which is meaningful performance above a random baseline on imbalanced data, but not high enough to use these predictions with clinical confidence.

The system predicts *stress-response pathway activation* as measured in cell-based reporter assays, which is a proxy for toxicity — not a direct measurement of organ-level or whole-organism toxicity. The three assays cover genotoxic, oxidative, and cellular stress; they do not cover every toxicity mechanism. A compound that clears all three checkpoints has not been comprehensively screened.

The applicability-domain check uses Tanimoto similarity to a sample of 300 training fingerprints, which is a fast heuristic rather than a rigorous statistical test. Out-of-domain warnings should be treated as a flag for reduced confidence, not as a definitive boundary.

## Future Work

Several natural extensions would make the system more useful. Training on additional Tox21 assays (or NR-pathway assays alongside the SR-pathway ones used here) would broaden mechanistic coverage. Adding Lipinski and ADMET descriptors to the feature set could improve model performance on drug-like compounds specifically. A proper conformal prediction layer would replace the heuristic confidence levels with statistically calibrated prediction intervals. Integrating RDKit's atom-contribution mapping with the SHAP fingerprint values could produce genuinely interpretable atom-level highlighted structure images. Moving from XGBoost to a graph neural network would potentially improve performance on structurally complex molecules, though at the cost of the SHAP interpretability story.

## Why This Is Different

Most toxicity prediction demos at student hackathons are single-model classifiers with a percentage score. ToxPathGuard is a multi-step audit: three independent models, each covering a distinct biological mechanism, combined by a rule-based coordinator that also checks whether the molecule looks like known toxicophores and whether it is structurally similar to anything the models were trained on. The result is not just "42% toxic" — it is a structured explanation of *which pathway* is flagged, *why* the model thinks so, *which chemical features* are responsible, and *how much to trust the answer*. That combination — predict, identify pathway, explain why, show where, warn when uncertain — is the design goal.

## What Judges Will See in the Demo

The demo starts with ethanol (a clear molecule) to show the normal Cleared output, then moves to benzo[a]pyrene (a well-known PAH carcinogen) to show all three agents firing at high confidence with a Confirmed Toxicophore label. From there it covers paracetamol as a borderline case where the structural alert fires but model scores are moderate (Alerted — Model Score Low), demonstrating that the system distinguishes between chemical pattern matches and model-based risk. The demo ends with an invalid SMILES to show graceful error handling. The full session, including the SHAP charts and the downloadable JSON report, fits comfortably inside three minutes.

---

## Why This Is Not Overengineered

The entire pipeline runs as a single Python process. There is no API server, no microservice architecture, no message queue, no containerisation, and no cloud dependency of any kind. The feature engineering is intentional minimalism: 1024 Morgan fingerprint bits plus six physicochemical descriptors, totalling 1030 features — enough to be expressive for this dataset scale, not so many that the SHAP explanations become uninterpretable. The three XGBoost models train in under two minutes on a consumer laptop. The "agentic" coordinator is a deterministic if-then policy function, not a language model or tool-calling agent, which means every output is traceable and reproducible. The system was deliberately scoped to be something one developer could build, understand, and demo confidently — which is the point of a hackathon prototype.
